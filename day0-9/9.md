## What I Learned
- ソケット = IPアドレス + ポート番号 の組み合わせで通信のエンドポイントを表す
- `ss` は `netstat` の後継、ソケットの状態(LISTEN / ESTABLISHED など)を一覧表示
- `nmap` は外部から能動的にポートをスキャンする(自分のサーバー以外には実行しない)
- 開いているポート = 攻撃面(attack surface)、不要なサービスは止めるのが原則
- `ufw` は `iptables`(netfilter)のフロントエンド、シンプルにファイアウォール設定できる
- `0.0.0.0` は全インターフェース公開、`127.0.0.1` はループバックのみ(同一マシンからしか到達不可)
- `localhost` 宛の通信はファイアウォールを通らない(loopback経由のため)
## Commands Used
```bash
# ポート・ソケットの確認
netstat                                   # 古いツール(非推奨、ssに置き換え)
netstat -l                                # LISTEN中のソケットのみ
ss -ltp                                   # TCPのLISTEN + プロセス情報
sudo ss -ltp                              # sudoでプロセス名まで表示
ss -tlnp                                  # 数値表示で速い(名前解決なし)

# nmap によるポートスキャン
sudo apt install nmap                     # インストール
nmap localhost                            # 自分のサーバーをスキャン
nmap -sV <ip>                             # サービス・バージョン特定(自分のサーバーのみ)

# ufw でファイアウォール
ufw                                       # ヘルプ表示
sudo ufw status                           # 現在の状態確認
sudo ufw status verbose                   # 詳細表示
sudo ufw allow ssh                        # SSH(22)許可 ※enableの前に必須
sudo ufw deny http                        # HTTP(80)拒否
sudo ufw allow http                       # HTTP(80)許可
sudo ufw enable                           # ufw有効化
sudo ufw disable                          # ufw無効化

# 動作確認
curl http://localhost                     # ローカルからのアクセステスト
sudo iptables -L                          # 実際のルールを確認(ufwが生成)

# その他
history                                   # 実行したコマンド履歴
```
## Where I Got Stuck
- `sudo ufw deny http` した後も `curl http://localhost` で繋がる → localhost(loopback)宛はファイアウォールを通らないため、外部から試さないと効果が確認できない
## Insights & Reflections
ファイアウォールは「外部からの通信」を制御するもので、loopback内の通信は素通りする。本当に効果を確認したければ別マシンや別ネットワークから検証する必要がある。`ss` でLISTEN中のポートを把握 → `ufw` で不要な入口を閉じる、という流れがアタックサーフェス管理の基本パターン。
